# Changelog - 2026-04-24

## Summary
Added the `stock_movements` (PrestaShop API: `stock_mvt`) resource as read-only, and harmonized all resource descriptions to include the technical API name in parentheses for improved readability (following the existing `Stock Available` pattern).

## Changes

- [nodes/PrestaShop8/types.ts] Added `stock_movements` resource (read-only: list/getById/search only) between `stock_availables` and `categories`. Appended `(api_name)` to the description of every resource (33 total) so that UI users see the actual PrestaShop API endpoint next to the human-readable name.
- [tests/unit/types.test.ts] Bumped expected resource count from 32 to 33, added a dedicated test for `stock_movements` read-only capabilities, and included `stock_movements` in the `limitedResources` list.
- [tests/ui/run-integration-tests.py] Added `stock_movements` entry (read-only) so the UI integration test suite covers the new resource.

## Notes
Tests: unit suite 191/191 pass; UI integration suite 89/93 pass (the 4 pre-existing failures are environmental — empty collections in the test PrestaShop store and the known `images/getById` limitation).

The `stock_movements` schema was already present in `resourceSchemas.ts` (auto-generated), and the fetch script (`scripts/fetch-prestashop-schemas.js`) already knew the alias mapping `stock_movements` ↔ `stock_mvt` — only the UI registration in `types.ts` was missing.

Observed side-note: when a user changes the `Resource` parameter while the node modal is open, n8n caches the previous resource's operations list. Workaround: close and reopen the node modal. This is an n8n UI behavior, not a node-level bug.
